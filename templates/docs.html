What It Is

Rubrica is a privacy-first, locally-run web application that automates the grading of handwritten student exams using Claude AI. Runs entirely on your laptop — no cloud database, no external services except the Anthropic API for vision grading. Student identities are never exposed to AI.

Built for GSIs and TAs who need to grade large volumes of handwritten exams quickly and consistently while maintaining FERPA-compliant anonymization during the AI grading step.

Core Privacy Architecture

The central design principle: student names never leave your machine.

  • Every exam is assigned a random 8-character anonymous ID (e.g., QX7KR2DA) at upload time.
  • Only the anonymous ID + exam page images + rubric are sent to Claude.
  • The name↔ID mapping lives solely in a local SQLite database on this machine.
  • Results are displayed by anonymous ID; names are only revealed on-demand via a toggle.
  • The exam cover page (page 0) is never sent during grading — only answer pages.

Private Mode

A Private Mode toggle (navbar button or dashboard card) hides all student identities across every page simultaneously — designed for use during presentations or screen-sharing without switching between views.

  • Names and SIDs on the name-assignment page become invisible (data still present and submitted correctly).
  • Cover page thumbnails and the full-page review modal blur when viewing exam pages.
  • Name and SID columns are suppressed on the Results page.
  • Roster preview hides all names; the Reveal Names button is removed.
  • Student name is hidden on individual report pages; the Reveal Name button is removed.
  • A red PRIVATE MODE badge appears in the navbar as a persistent reminder.
  • Disabling Private Mode from any page instantly restores all normal functionality.

Data Flow — End to End

1
Batch PDFs uploaded. Up to 3 batch PDFs at once, each with its own batch number. All student exams concatenated within each file, in order.
2
Split by fixed page count — entirely in memory. All batches split and merged into one review session in a single pass. No file handles held open, no Claude API calls, no cost. A perceptual hash (8×8 phash) of each cover page is computed locally to flag any exam whose layout differs structurally from the majority.
3
Name extraction + review page. Ollama ({{ ollama_model }}) reads each cover page (page 0) locally in a background thread and pre-fills student name and SID with live progress. Extraction is opt-in (skip on CPU-only machines) and can be aborted mid-run — names extracted so far are preserved. A thumbnail and full-page modal viewer let you verify all pages are attached and correct any misreads before confirming. Roster fuzzy-match available if a class list has been uploaded. A persistent Scan nav tab allows returning to an in-progress session from anywhere in the app.
4
Saved to SQLite. Each exam gets a random anon_id. The student name is stored locally only — never included in any API call.
5
Grade All triggered. A thread pool (5 concurrent workers) is spawned so exams grade in parallel. Already-graded exams are skipped automatically. A live progress bar polls /grade-all/status every 3 seconds.
6
Per-exam grading. Pages 1+ rendered as PNG at 2x scale, contrast-enhanced (2x boost for faint pencil legibility), and sent to Claude Sonnet as images. The rubric PDF is sent as a native base64 document block (preserves tables and charts). Cover page always skipped. If Claude returns malformed JSON, the call retries once automatically.
7
Claude returns structured JSON. Question-level scores, feedback, total, letter grade, and overall summary. Post-processing runs in Python: sub-part consolidation, feedback sanitizer, score recalculation, and feedback specificity enforcement (vague feedback refined via a text-only follow-up call). Saved to the DB under anon_id only.
8
Boundary re-grade. If the final score falls within +/-1.5% of a letter grade threshold (90/80/70/60), the exam is automatically re-graded in a second independent pass. If the two passes disagree on the letter grade, scores are averaged. Both passes are stored in the audit trail.
9
Results page. Grades shown by anon_id. Toggle reveals student names locally. Collapsible detail rows allow per-question score adjustment (saved via AJAX, no page reload). Summary CSV exports one row per student; Detailed CSV exports one row per student with a column per question (earned, max, feedback). Per-student printable report opens in a new tab — clean print layout with score card, progress bar, color-coded question breakdown, and overall feedback.

Key Features

Upload & Splitting

  • Upload up to 3 batch PDFs at once — each with its own batch number; version and pages-per-exam are shared.
  • All batches split and merged into one review session in a single pass (no API cost).
  • Single-exam upload: set pages-per-exam equal to the exam page count.
  • Cover page consistency check — after split, a perceptual hash (8×8 phash, Hamming distance threshold 15/64) compares all cover pages locally. Outliers get a yellow thumbnail border and ⚠ Cover badge. Runs on-device, no API.
  • Name assignment page shows a cover-page thumbnail per exam, batch number column, and a full modal page viewer.
  • Ollama ({{ ollama_model }}) reads each cover page locally in a background thread to pre-fill student name and SID — no API cost, no data leaves the machine. Opt-in checkbox to skip on CPU-only machines. Abortable mid-run via Stop button.
  • A Scan nav tab appears between Upload and Exams whenever an unsaved session exists — survives page navigation and app restarts.
  • Roster fuzzy-match confirms identity against an uploaded class list (local only, no API).
  • Names and SIDs can be hidden during review with a toggle — values are always preserved in the form and submitted correctly.
  • Cover page blurs automatically in the full-page review modal when Private Mode is active.
  • Names optional at upload time; editable anytime via each exam's detail page.

Rubric Management

  • Upload rubric as PDF or DOCX, or type it manually.
  • PDF rubrics sent natively to Claude as base64 document blocks — tables, charts, and formatting are fully preserved (not lost to text extraction).
  • Multiple rubric versions ({{ rubric_versions|join(', ') if rubric_versions else 'e.g. A, B' }}) for different exam variants.
  • Rubrics stored permanently on disk; can be deleted or replaced at any time.

Vision Grading Engine

  • Exam pages rendered as PNG and sent directly to Claude Sonnet - works on handwritten answers with no OCR pre-processing required.
  • Faint pencil handling: 2x contrast enhancement applied to every page image before sending to Claude; prompt instructs careful examination before concluding a question is blank.
  • Parallel grading: 5 concurrent workers grade exams simultaneously via ThreadPoolExecutor - batch grading runs ~5x faster with the same API cost.
  • Deterministic output: temperature=0.0 ensures identical exams receive identical scores across grading runs.
  • Feedback sanitizer: post-processing strips AI deliberation language ("wait", "actually", "let me re-read") and replaces em/en dashes with regular dashes in all feedback - deterministic regex, runs on every grading response.
  • JSON retry: if Claude returns malformed JSON, the grading call retries once automatically (max 2 attempts, hard-capped).
  • Batch resume: re-running Grade All skips already-graded exams before calling the API - no duplicate API spend.
  • Boundary re-grading: exams scoring within +/-1.5% of a letter grade threshold (90/80/70/60) are automatically re-graded in a second independent pass. If the two passes disagree on the letter grade, scores are averaged. Full audit trail stored per exam.
  • Feedback specificity enforcement: vague feedback ("Good work", "Incorrect", or under 8 words) is detected post-grading and refined via a text-only API call requiring rubric-anchored, student-specific justification.
  • Score recalculation: sub-part consolidation (e.g., Q3a + Q3b into Q3) and normalization run in Python after the API response, ensuring arithmetic consistency independent of model output.
  • Cover page (page 0) always skipped; answer pages only sent to Claude.
  • Background thread with live progress bar - browser stays usable during long grading runs.
  • Persistent logging: all grading successes, failures, and retries written to data/grading.log - survives app restarts.
  • Filter by version and batch for targeted re-grading.
  • Per-exam re-grade and clear-grade buttons; bulk clear-all-grades option.

Multiple Choice Accuracy

System prompt includes explicit MC instructions to improve accuracy on handwritten exams:

  • Options listed vertically (A top → D bottom).
  • Claude reads the actual letter shape — B has two bumps on the right; D has one large curve.
  • No partial credit on MC; handles ambiguous double-marks by using the most deliberate one.

Per-Exam Scan Page

  • Page-by-page viewer for every uploaded exam.
  • Cover labeled "not sent to Claude"; answer pages labeled "sent to Claude for grading."
  • Cover page blurs automatically when Private Mode is active.
  • Editable student name, re-grade, and clear-grade controls per exam.

Results & Export

  • Results page filterable by version and batch.
  • Student names hidden by default, revealed with a local-only toggle.
  • Private Mode suppresses all name/SID columns across Results, Roster, and Reports simultaneously.
  • Grade adjustment: expand any row's Details panel to edit per-question earned points inline; changes update the summary row live and save via AJAX.
  • Reviewed checkmark: per-exam toggle to track review progress; reviewed rows get a green highlight that persists across sessions and page reloads.
  • Summary CSV: one row per student — Anon ID, Name, SID, Version, Batch, Score, Percentage, Letter Grade, Overall Feedback.
  • Detailed CSV: same fixed columns plus three columns per question (Earned, Max, Feedback) — pads with empty strings for questions absent from a given exam.
  • Per-student report: printable HTML page per student with score card, progress bar, color-coded question breakdown, and overall feedback. Print-ready via browser.
  • Score normalization: if Claude's reported total_possible is within 0.5 of a whole number, both totals are scaled so scores display cleanly. Earned is hard-capped at possible.

Analytics

  • Grade distribution histogram and grade band donut chart.
  • Summary stats: mean, median, standard deviation, range, pass rate.
  • Per-question average score bar chart — color-coded by performance tier.
  • Automatic alert for questions where average score falls below 60%.
  • All charts filterable by version and batch.

Validation and Research

Rubrica's grading pipeline and audit methodology are grounded in established psychometric standards and recent AI grading research. The audit system uses Gemini 3.1 Pro as a cross-family reference scorer to validate production grades assigned by Claude Sonnet 4.6, with Claude Opus 4.6 available as a within-family fallback. A blind validation analyst (with no knowledge of pipeline internals) independently evaluates inter-rater reliability, bias, and feedback quality.

Audit Protocol

The audit system (audit_grader.py) re-grades stratified random samples under identical conditions (same rubric, same anonymized pages, temperature 0.0) and computes inter-rater reliability metrics at the individual question level. All per-question metrics are stratified by rubric version, since different versions have different questions and point allocations. Reports are generated for faculty review: generate_audit_report.py (ETS benchmark report), generate_review_checklist.py (flagged exam review workflow), and generate_presentation_bundle.py (combined PDF with title page, TOC, and all validation documents).

Research-Backed Safeguards

  • Boundary re-grading addresses the known concentration of inter-rater disagreements at letter grade thresholds (Williamson et al., 2012).
  • Feedback specificity enforcement responds to research showing AI feedback outperforms human feedback on metacognitive dimensions when citing specific student work (Nazaretsky et al., 2026, Journal of Computer Assisted Learning).
  • 2x contrast enhancement mitigates handwriting quality bias documented in vision LLM grading (arXiv:2601.16724).
  • Deterministic scoring (temperature 0.0) eliminates random variation, a baseline requirement for reliable assessment (Fleiss, 1981).

Database Schema

-- Rubrics: one per exam version; stores path to original PDF/DOCX on disk
rubrics (
  id               INTEGER PRIMARY KEY,
  version          TEXT,          -- "A", "B", etc.
  content          TEXT,          -- extracted text (fallback / display)
  rubric_file_path TEXT,          -- path to original file → sent natively to Claude
  created_at       TIMESTAMP
)

-- Exams: the name↔ID mapping never leaves this table
exams (
  id           INTEGER PRIMARY KEY,
  anon_id      TEXT UNIQUE,   -- random 8-char ID — the only identifier sent to Claude
  student_name TEXT,          -- stored locally ONLY, never sent to Claude
  student_sid  TEXT,          -- student ID number — stored locally ONLY
  version      TEXT,          -- exam version (A, B, …)
  batch        INTEGER,       -- batch number for grouping
  file_path    TEXT,          -- path to individual exam PDF on disk
  uploaded_at  TIMESTAMP,
  graded_at    TIMESTAMP,
  grade_data   TEXT,          -- JSON blob: scores, feedback, totals, letter grade
  reviewed     INTEGER        -- 0/1: instructor review checkmark (persists across sessions)
)

-- Roster: class list for fuzzy name/SID matching at upload time (local only)
roster (
  id         INTEGER PRIMARY KEY,
  first_name TEXT,
  last_name  TEXT,
  sid        TEXT
)

What It Is Not

  • Not a cloud service — all data (DB, PDFs, rubrics) stays on your machine.
  • Not a real-time system — grading is batch, runs in a background thread.
  • Not OCR-dependent — grading sends page images directly; no text extraction step required.
  • Not identity-aware at the AI layer — Claude never sees a student name during any grading call.
